# Логика формирования шапки коммерческого предложения

## Описание

Система автоматически определяет тип получателя (физическое лицо или организация) и соответствующим образом формирует шапку коммерческого предложения.

## Типы получателей

### 1. Физическое лицо

**Определение:**
- В тексте сообщения присутствуют ключевые слова: `физику` или `для физика` (регистронезависимо)
- **ИЛИ** отсутствуют данные организации в векторной БД

**Формат шапки:**
```
┌──────────────────────────┬──────────────────────────┐
│ ЛЕВЫЙ СТОЛБЕЦ            │ ПРАВЫЙ СТОЛБЕЦ           │
├──────────────────────────┼──────────────────────────┤
│ «24» декабря 2025        │ г. Москва                │
│                          │                          │
└──────────────────────────┴──────────────────────────┘
```

### 2. Юридическое лицо (организация)

**Определение:**
- Найдены данные организации в векторной БД Qdrant
- **И** нет ключевых слов физического лица

**Формат шапки:**
```
┌──────────────────────────┬──────────────────────────┐
│ ЛЕВЫЙ СТОЛБЕЦ            │ ПРАВЫЙ СТОЛБЕЦ           │
├──────────────────────────┼──────────────────────────┤
│ «24» декабря 2025        │ «Энерком Альянс»         │
│ г. Москва                │ ИНН 4028058015           │
│                          │ КПП 402901001            │
│                          │ Генеральному директору   │
│                          │ Ахметшину Альберту       │
│                          │ Ренатовичу               │
└──────────────────────────┴──────────────────────────┘
```

## Реализация

### Поток данных

```
Сообщение пользователя
        ↓
Анализ ключевых слов (физику/для физика)
        ↓
Поиск компании в Qdrant (если не физлицо)
        ↓
Получение данных из PostgreSQL
        ↓
Формирование offerData.userText + offerData.clientCompany
        ↓
Рендер PDF с правильным форматом шапки
```

### Код в index.js

```javascript
// Поиск компании клиента в векторной БД и PostgreSQL
if (offerData?.products?.length) {
  // Сохраняем исходный текст для анализа типа получателя
  offerData.userText = userText;

  // Ищем компанию в Qdrant
  const searchResults = await searchCompaniesInQdrant(userText, 3);

  if (searchResults.length > 0) {
    const bestMatch = searchResults[0];

    // Получаем полные данные из PostgreSQL
    const result = await client.query('SELECT * FROM company WHERE id = $1',
      [bestMatch.company.id]);

    if (result.rows.length > 0) {
      // Добавляем данные компании клиента
      offerData.clientCompany = {
        name_short: clientCompany.name_short,
        name_full: clientCompany.name_full,
        inn: clientCompany.inn,
        kpp: clientCompany.kpp,
        ceo_name: clientCompany.ceo_name,
        ceo_name_dative: clientCompany.ceo_name_dative
      };
    }
  }
}
```

### Код в offerSheet.js

```javascript
// Определяем тип получателя
const userText = offerData.userText || '';
const isIndividual = /\b(физику|для\s+физика)\b/i.test(userText);
const clientCompany = offerData.clientCompany || {};
const hasCompanyData = clientCompany.inn || clientCompany.name_short ||
                       clientCompany.name_full;

// Левый столбец
if (hasCompanyData && !isIndividual) {
  // Дата + "г. Москва"
  doc.text(formattedDate, ...);
  doc.text('г. Москва', ...);
} else {
  // Только дата
  doc.text(formattedDate, ...);
}

// Правый столбец
if (isIndividual || !hasCompanyData) {
  // Только "г. Москва"
  doc.text('г. Москва', ...);
} else {
  // Полные реквизиты компании
  doc.text(companyName, ...);
  doc.text(`ИНН ${companyInn}`, ...);
  doc.text(`КПП ${companyKpp}`, ...);
  doc.text('Генеральному директору', ...);
  doc.text(ceoDative, ...);
}
```

## Ключевые слова для определения физического лица

- `физику` - дательный падеж (кому?)
- `для физика` - родительный падеж (для кого?)

**Регулярное выражение:** `/(физику|для\s+физика)/i`

**Важно:** Границы слов `\b` не используются, так как они не работают с кириллицей в JavaScript.

## Приоритет определения типа

1. **Высший приоритет:** Наличие ключевых слов физического лица → физлицо (поиск в Qdrant не выполняется)
2. **Средний приоритет:** Поиск в Qdrant с проверкой релевантности:
   - Релевантность >= 50% → юрлицо (используются данные компании)
   - Релевантность < 50% → физлицо (низкая уверенность в совпадении)
3. **Низший приоритет:** Ничего не найдено → физлицо (по умолчанию)

## Порог релевантности

**Минимальный порог:** 50%

Если при поиске в Qdrant лучшее совпадение имеет релевантность < 50%, система считает это **физическим лицом**.

### Примеры:

| Запрос | Релевантность | Результат |
|--------|---------------|-----------|
| "КП на зарядную станцию" | 26.8% | Физлицо (нет упоминания компании) |
| "КП для Энерком" | 50.6% | Организация (частичное совпадение) |
| "КП для компании Энерком Альянс" | 68.3% | Организация (точное совпадение) |

### Обоснование порога 50%:

- **< 50%:** Низкая уверенность - возможно случайное совпадение слов
- **>= 50%:** Средняя/высокая уверенность - явное упоминание компании
- **> 70%:** Высокая уверенность - точное совпадение названия

## Оптимизация

Если в тексте обнаружены ключевые слова физического лица, поиск компании в Qdrant **не выполняется**. Это экономит:
- Время выполнения (~500-1000мс на запрос к OpenAI для эмбеддинга)
- Стоимость API вызовов к OpenAI
- Нагрузку на Qdrant
